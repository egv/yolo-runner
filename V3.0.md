# Version 3.0 of yolo runner

This update will fully rework system architecture.


## Main arch changes

Right now there are yolo agent that manages tasks and spawns runners. We need to redo it this way

task manager <-> mastermind orchestrator <-> runners

### Task manager

Task manager, well, it manages tasks. It gathers them into one inbox from different sources (linear, github, tk, br, st etc.) as well as persists changes that are made to the tasks back. It can link tasks from different systems inside itself, and those are not only link that statest that this task in system a is that task in system b, but also it should link them in term of blocking, following etc. It should store all of that locally somewhere. Interface for tasks can be two way (like linear API) and one way (like tasks from telegram messages). Probable we will still need to update one-way tasks by posting some kind of messages, but this is what we still need to think about. 

Main function of the task manager is that at any given time it is able to give mastermind a list of tasks that needs to be done at this given moment. 

All external interactions (like linear or github) should be preferrably described by skills that utilize CLI tools rather than be implemented in code, so one can easily add it's own intergration if needed. Surely skill should describe how to do all required actions, and as a part of an implementation process here we should figure out that minimal, but sufficent set of actions.

#### Preferred order supported backends 
- github (gh for basic things and gh api for api calls to sent task dependencies)
- linear (we have api support right now, but it is untested so I would look into something like https://github.com/czottmann/linearis or https://github.com/schpet/linear-cli, the second one looks better as for me)
- br (https://github.com/Dicklesworthstone/beads_rust), it looks like fire to me
- skill support and then any other task tracker that is implemented through the corresponding skill

#### Thoughts:
- We should probably learn how to sync all with br and use it as a local underlying storage (at least consider doing so).
- This looks like a completely separate system from the rest, and it should communicate with it using some kind of rpc or cli interface.
- Sync mode should be batched and ran once in a while. We should have some kind of confing files that show how and with what to sync tasks.
- We should have CLI, WEB and TUI for different reasons. CLI for this be used in AI agents. TUI and WEB for users. 
- this should be done in Rust. Just want it that way.
- WEB and TUI should support kanban-like board views, filtering etc., keyboard shortcuts, all the bells and whistles
- Ratatui (https://github.com/ratatui/ratatui) for TUI.

### Mastermind

This entity gets tasks, validates them to see if they have comprehonsive defenition of done and acceptance criteria, somehow signals user if there are updates needed for a task (by posting comments or sending user a message, all that done through a task manager), and, if the task looks ok, it posts it to the execution queue. After that it communicates with the agent that picks that task up and updates the task in task manager accordingly.

For all tasks clarification and validation purposes mastermind should also use special kind of runners. This brings us to the fact that this thing should have several predefined workflows for different types of tasks, and those workflows should be described as graphs and be easily configured. I want it to be that way so mastermind can use smaller, probably local model and it's process to me more deterministic. This thing should parse graph from some human-readable format like YAML, then create some kind of state machine for that inside itself, run it and do the job.

This thing should be running as daemon and be accessible via some kind of RPC or API, as well as it should be able to query external task manager systems. Yes, systems. Yes, I think that there should be more that one task manager available. Communications there should be both push and pull, so mastermind should be able to both query task manager periodically and be queried by taskmaster in case network connectivity allows only outgoung connection for a given task master.


### Runner

This is an entity that provides interface to different AI agents like opencode, codex cli, kimi cli, pi, claude etc. to execute tasks from the mastermind. We should note that task in this context is not the same as the task in context of a task manager part. Here task is a fully described job that can be formally verified using tests or LLM as a judge or other method, that gives formal and definitiv answer if it is ready or not.

Runner should have some kind of tags or skills, or properties so they can be matched with the ones that mastermind marks tasks with to be sure that correct runner picks correct task. This is important because some runners may lack cerain tools like image processing or access to hosts/repos, so this should be used to pick tasks. 

When executing it's task runner should post updates on task's status for mastermind to see and react on that.

#### Codex runner

We should switch to codex app server instead of parsing is's output. See https://developers.openai.com/codex/app-server/ for more details on a protocol

#### API connection to agent

Since all our main agents (codex, opencode and pi) support some sort of RPC we should do adaptors for them and then one core that will process all interactions like granting permissions etc.

#### Gemini cli ACP support

We need to add another coding backend alongside with codex, opencode, claude and kimi. We should also have ability to assign different taks to different agents. Probably we should be able to describe all agentic backends in some kind of uniform way so user will be able their own agents as they want, and the runner's job will be to communicate with those and translate their state to the events bus for mastermind to react upon.

#### ACP vs. API

ACP is just another RPC type, so everything should work just fine by implementing another adapter.

## More thoughts

This state machine thing that I've described in mastermind section should really be used in all three parts, but with different graph of course. This will give us a lot of flexibility and ability to extend system for the ways we have not even thought of.

## Implementation order

This is a completely new stuff. We start fresh. We start in Rust.

- we should pick the correct lib for state machine/agentic graph management. I think of https://github.com/mdeloof/statig for state machine and https://github.com/0xPlaygrounds/rig or https://github.com/liquidos-ai/AutoAgents for nodes where we need real LLM calls. This all should be configured with human-readable YAML config, parsed and executed. It should be made a crate and used in all other parts of this system.
- we should separate task manager into different repo and start work with it in parallel. For now, mastermind and runner should be in one repo, but separate commands that we will probably split later.
- implement generic RPC adaptor for runners
  - implement codex app server based on top of it
  - implement opencode api/rpc adapter (instead of ACP protocol)
  - implement pi RPC adapter (https://github.com/badlogic/pi-mono/tree/main)
- implement basic mastermind
  - rpc to add task
  - invoking runner with task
- implement rest of the mastermind
